# 综合新闻 APP - 研发流程与开发规范指南

## 1. 目的与概述
为了保证“综合新闻 APP”项目在前后端分离架构下的开发质量、团队协作效率，以及代码的可维护性，特制定本《研发流程与开发规范指南》。本指南覆盖了从代码版本管理、分支模型、代码规范到测试与部署的全生命周期最佳实践。

---

## 2. 版本控制与分支模型 (Git Workflow)
本项目采用精简版的 **Git Flow** 工作流，确保开发、测试与生产环境的代码隔离。

### 2.1 核心分支说明
| 分支名称 | 描述 | 保护策略 |
| :--- | :--- | :--- |
| `main` / `master` | **生产环境分支**。该分支的代码必须是经过全面测试、随时可发布上线的稳定版本。 | 限制直接 Push，仅允许通过 Merge Request (MR) 合并。 |
| `develop` | **开发主分支**。所有新功能开发完成并自测通过后，合并到此分支。 | 限制直接 Push，仅允许通过 MR 合并。 |

### 2.2 临时分支说明
*   **功能分支 (`feature/*`)**：
    *   用途：开发新功能（如 `feature/user-login`, `feature/news-feed`）。
    *   来源：从 `develop` 分支切出。
    *   去向：开发完成后，发起 MR 合并回 `develop` 分支。
*   **修复分支 (`hotfix/*` 或 `bugfix/*`)**：
    *   用途：修复线上或测试环境的紧急 Bug（如 `hotfix/login-crash`）。
    *   来源：如果是线上 Bug，从 `main` 切出；如果是测试阶段 Bug，从 `develop` 切出。
    *   去向：修复后合并回来源分支（若修复了 `main`，则也必须同步合并到 `develop`）。

### 2.3 Commit Message 规范
提交信息必须清晰描述本次代码变更的目的，采用 [Angular 规范](https://github.com/angular/angular.js/blob/master/DEVELOPERS.md#commits)：
格式：`<type>(<scope>): <subject>`
*   `feat`: 新功能 (Feature)
*   `fix`: 修复 Bug
*   `docs`: 文档修改
*   `style`: 代码格式修改（不影响逻辑）
*   `refactor`: 代码重构（既不修复 Bug 也不添加新功能）
*   `test`: 增加或修改测试用例
*   `chore`: 构建过程或辅助工具的变动
**示例**：`feat(user): 新增用户手机号验证码登录接口`

## 3. 代码规范 (Coding Standards)

### 3.1 前端规范 (UniApp / Vue3)
1.  **框架与语法**：统一使用 Vue 3 的 `<script setup>` 组合式 API (Composition API)。
2.  **Linting & Formatting**：
    *   必须接入 `ESLint` 和 `Prettier`。
    *   保存时自动格式化（通过 IDE 插件如 Volar/Prettier 配置）。
3.  **命名约定**：
    *   组件文件名：`PascalCase.vue` (如 `NewsCard.vue`)。
    *   变量与方法：`camelCase` (如 `fetchNewsList`)。
    *   CSS 类名：采用 BEM 命名法，使用 `-` 分隔 (如 `news-card__title`)。
4.  **注释与日志**：
    *   **严格遵守《kxmall-app-ui 代码注释与 Console 输出规范》**。
    *   核心业务逻辑、复杂状态判断必须写注释。
    *   生产环境禁用 `console.log`，调试使用 `console.info / warn / error`。

### 3.2 后端规范 (Node.js)
1.  **架构分层**：严格遵循 `Controller` -> `Service` -> `Model/DAO` 三层架构，禁止在 Controller 中直接写复杂的业务逻辑或 SQL。
2.  **Linting & Formatting**：必须接入 `ESLint` 和 `Prettier`。推荐使用 TypeScript 以获得更好的类型检查。
3.  **命名约定**：
    *   文件命名：小写，使用 `-` 分隔 (如 `user-controller.js`)。
    *   变量与方法：`camelCase`。
    *   类名：`PascalCase`。
4.  **异常处理**：
    *   禁止在 Controller 中使用裸露的 `try-catch`，应使用全局异常捕获中间件（如 Express 的 Error-handling Middleware）。
    *   业务异常应抛出自定义的 `BusinessError`，由全局中间件统一格式化为标准 JSON 响应。

## 4. 联调与测试规范 (Testing & Integration)

### 4.1 前后端联调流程
1.  **契约先行**：后端开发前，必须先在《API接口规范说明书》或 API 管理工具（如 Apifox、YApi）中定义好接口并提供 Mock 数据。
2.  **前端并行**：前端基于 Mock 数据进行页面和逻辑开发。
3.  **真实联调**：后端接口开发完毕并在本地（或联调环境）自测通过后，前端切换 Base URL，进行真实数据联调。

### 4.2 测试规范
1.  **单元测试 (Unit Testing)**：
    *   后端：核心业务逻辑（如计费、复杂权限校验）建议编写单元测试（使用 Jest 或 Mocha）。
    *   前端：核心的公共组件和 Utils 函数建议编写单元测试。
2.  **代码审查 (Code Review)**：
    *   合并 MR 前，必须进行 Code Review。
    *   审查重点：代码是否符合规范、是否存在潜在 Bug、是否缺少必要的注释、是否有安全漏洞（如 SQL 注入风险）。

## 5. 部署与运维规范 (Deployment & DevOps)

### 5.1 环境划分
*   **本地环境 (Local)**：开发人员自己的机器，连接本地或共享的开发数据库。
*   **测试环境 (Test/QA)**：用于前后端联调和功能测试，对应 `develop` 分支。
*   **生产环境 (Production)**：正式面向用户的环境，对应 `main` 分支。

### 5.2 配置文件管理
*   **环境变量**：使用 `.env` 文件管理不同环境的配置（如数据库密码、API 密钥）。
*   **安全要求**：`.env` 文件**绝对禁止**提交到 Git 仓库，必须添加到 `.gitignore` 中。仓库中应提供一个 `.env.example` 作为配置模板。

### 5.3 自动化构建与部署 (CI/CD) - 可选
*   推荐使用 GitHub Actions / GitLab CI / Jenkins 实现自动化构建和部署。
*   流程：代码 Push -> 触发 Lint 检查 -> 运行单元测试 -> 构建镜像 (Docker) -> 自动部署到测试服务器。
