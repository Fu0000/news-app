# 综合新闻 APP - 前后台页面架构与交互设计文档

## 1. 概述 (Overview)
本项目分为两个核心前端工程：
1.  **用户端 (C端 APP)**：面向普通用户，提供资讯浏览、互动、内容发布等功能，采用 `UniApp` (Vue 3) 开发，支持编译为 H5、微信小程序及原生 App。
2.  **管理后台 (B端 Admin)**：面向运营人员和管理员，提供内容审核、用户管理、数据统计等功能，采用 `Vue 3` + `Element Plus` 开发（SPA 单页应用）。

本指南结合互联网 C 端内容平台（如今日头条）和 B 端中后台（如 Ant Design Pro）的最佳实践，详细规范页面的架构、路由、组件复用与状态流转。

## 2. 用户端 (C端 APP) 页面架构设计

### 2.1 页面层级与路由规划 (Page Routing)
C端采用经典的“底部导航栏 (TabBar) + 沉浸式子页面”的层级结构。

| 层级 | 页面名称 | 路由路径 | 核心组件与交互说明 | 状态依赖 |
| :--- | :--- | :--- | :--- | :--- |
| **TabBar** | **首页 (资讯流)** | `/pages/index/index` | 顶部频道横滑 Tab (`scroll-view`)；<br>中间无限下拉瀑布流 (`onReachBottom`)；<br>卡片支持单图/三图/视频模式切换。 | 全局分类列表缓存 |
| **TabBar** | **视频 (沉浸流)** | `/pages/video/index` | 类似抖音的上下滑动视频流 (`swiper` + `video`)；<br>视口内自动播放，滑出自动暂停。 | - |
| **TabBar** | **发布 (UGC)** | `/pages/publish/index` | 半屏弹窗或独立页选择发布类型；<br>富文本编辑器、九宫格图片上传组件。 | **强制登录拦截** |
| **TabBar** | **我的 (个人中心)** | `/pages/user/index` | 顶部用户信息与统计数据（获赞、关注）；<br>中间金刚区功能入口；底部设置菜单。 | 登录状态 (`Token`) |
| **二级页** | **图文详情页** | `/pages/detail/article` | 顶部透明导航栏（随滚动渐变）；<br>中间富文本解析 (`mp-html`)；<br>底部吸底交互条（输入框、点赞、收藏）。 | 详情数据、点赞状态 |
| **二级页** | **登录/注册** | `/pages/login/index` | 手机号一键登录/验证码登录切换；<br>动画表单校验；登录成功后返回上一页。 | - |
| **二级页** | **全局搜索** | `/pages/search/index` | 顶部搜索框带防抖 (Debounce)；<br>中间历史搜索标签 (`Tag`) 与热搜榜单。 | 本地存储 (`Storage`) |

### 2.2 前端状态管理流转 (State Management)
使用 `Pinia` 管理跨页面的核心状态，避免深度组件传递 (Prop Drilling)：
1.  **用户状态 (`userStore`)**：
    *   **State**: `token` (鉴权凭证), `userInfo` (昵称、头像), `isLogged` (计算属性)。
    *   **Action**: `login()` (调用接口并存入 Storage), `logout()` (清除 Token 与 Storage)。
2.  **系统状态 (`appStore`)**：
    *   **State**: `categoryList` (新闻分类频道，应用启动时拉取一次并持久化，避免首页每次重新请求)。

## 3. 管理后台 (B端 Admin) 页面架构设计

### 3.1 页面布局与路由规划 (Admin Routing)
B端采用“左侧侧边栏菜单 (Sidebar) + 顶部顶栏 (Header) + 右下内容区 (Main)”的经典布局。

| 模块 | 页面名称 | 路由路径 | 核心组件与交互说明 | 权限要求 |
| :--- | :--- | :--- | :--- | :--- |
| **基础** | **登录页** | `/login` | 账号密码登录表单，滑块/图形验证码防刷。 | 无 |
| **看板** | **数据大盘** | `/dashboard` | ECharts 折线图（日活趋势）、饼图（内容分类占比）、核心指标数据卡片。 | 基础权限 |
| **内容** | **文章管理** | `/content/article` | 顶部高级搜索表单；<br>中间数据表格 (Data Table)；<br>操作列支持“下架/推荐/查看详情”。 | 内容运营 |
| **内容** | **视频审核** | `/content/video` | 视频列表与播放弹窗；<br>支持批量审核通过/驳回操作。 | 审核专员 |
| **互动** | **评论管理** | `/interaction/comment`| 评论内容列表，支持敏感词高亮标红；<br>一键删除违规评论并连带处理子回复。 | 内容运营 |
| **用户** | **用户列表** | `/user/list` | 用户信息表格，支持账号封禁/解封操作。 | 管理员 |

### 3.2 B端前端架构最佳实践
1.  **动态路由与权限控制 (RBAC)**：
    *   用户登录后，前端通过接口获取用户的角色 (Role) 和权限菜单树。
    *   前端使用 `Vue Router` 的 `addRoute` 动态挂载用户有权访问的页面路由。
    *   未授权的菜单在侧边栏自动隐藏，非法输入 URL 时拦截至 `403` 页面。
2.  **表格与表单封装**：
    *   高度封装 `ProTable` 组件，内置分页、查询、重置逻辑，减少 CRUD 页面的重复代码。
    *   表单提交前强制进行前端校验（如手机号正则、必填项），减轻后端压力。

## 4. 后后台服务架构与数据流转 (Backend Architecture & Data Flow)

结合 Node.js (NestJS / Express) 体系，后端服务采用分层架构设计，确保高内聚低耦合。

### 4.1 后端技术分层架构 (Layered Architecture)

```mermaid
graph TD
    Client[客户端 (APP/H5/Admin)] -->|HTTP/HTTPS + JSON| Gateway[Nginx / 负载均衡]
    
    subgraph Node.js 业务服务层
        Gateway -->|路由分发| Router[Router/Controller 层<br>参数校验、组装响应]
        Router -->|调用| Middleware[Middleware 中间件<br>鉴权 JWT、日志打印、异常捕获]
        Router -->|业务逻辑| Service[Service 层<br>核心业务规则、事务控制]
        Service -->|数据持久化| DAO[DAO/Repository 层<br>ORM/Prisma 模型映射]
    end

    subgraph 基础设施层
        DAO -->|读写| MySQL[(MySQL 8.0<br>核心业务数据)]
        Service -->|读写/缓存| Redis[(Redis<br>验证码、热点新闻、点赞计数)]
        Client -->|前端直传| OSS[阿里云 OSS / MinIO<br>图片视频文件]
    end
```

### 4.2 核心数据传输与逻辑流转 (Data Logic Flow)

#### 场景一：C端用户“浏览并点赞文章”的流转过程
这是一个典型的高并发读写场景，体现了 Redis 缓存的最佳实践：
1.  **读取文章详情**：
    *   C端发起 `GET /api/v1/news/1001`。
    *   `Service` 层先查 `Redis` (如 `article:1001:detail`)，若命中则直接返回 JSON。
    *   若未命中（缓存穿透/失效），则穿透到 `DAO` 层查 `MySQL`，组装数据后写入 `Redis` 并设置过期时间，最后返回给 C端。
2.  **执行点赞操作**：
    *   C端发起 `POST /api/v1/interaction/like` (带 JWT Token)。
    *   `Middleware` 校验 Token 有效性并解析出 `userId`。
    *   `Service` 层校验该文章是否存在。
    *   **异步优化**：为了抗住高并发点赞，不直接写 MySQL。先在 `Redis` 中对该文章的点赞数 `INCR`，并在 Redis 的 Set 集合中记录该用户的点赞行为。
    *   **定时落盘**：Node.js 后台通过定时任务 (Cron Job) 或消息队列，定期将 Redis 中的点赞增量数据批量写入 MySQL 的 `user_interactions` 和 `articles` 表，实现最终一致性。

#### 场景二：C端用户“发布带图文章”的流转过程
这是一个典型的“前端直传 + 后端鉴权存储”最佳实践，避免 Node.js 服务器成为大文件上传的瓶颈：
1.  **获取上传凭证**：
    *   C端准备上传图片前，先调用 `GET /api/v1/common/upload/token` 获取 OSS 的临时 STS 签名。
2.  **前端直传文件**：
    *   C端拿到签名后，直接通过 `uni.uploadFile` 将图片上传至阿里云 OSS。
    *   OSS 返回图片的公网 URL（如 `https://oss.com/img1.jpg`）。
3.  **提交业务数据**：
    *   C端携带文章标题、正文（含图）以及刚获取到的封面图 URL 数组，发起 `POST /api/v1/content/publish`。
    *   Node.js `Controller` 接收并校验必填字段。
    *   `Service` 层处理防 XSS 注入（对 HTML 正文进行消毒），构建实体对象。
    *   `DAO` 层将记录 `INSERT` 到 MySQL 的 `articles` 表。
    *   响应 C端“发布成功”。

#### 场景三：B端运营“封禁违规用户”的流转过程
1.  **提交封禁指令**：
    *   B端管理员在表格中点击封禁，发起 `PUT /api/v1/admin/user/ban`。
    *   `Middleware` 拦截请求，校验 JWT Token，并检查该 Token 绑定的角色是否具有 `admin` 权限 (RBAC 鉴权)。
2.  **状态更新与缓存清理**：
    *   鉴权通过后，`Service` 层将 MySQL 中该用户的 `status` 字段更新为 `2` (封禁)。
    *   **关键步骤**：强制将该用户在 Redis 中缓存的 JWT Token 吊销（或拉入黑名单 Blacklist），确保其移动端 APP 的现有登录态立即失效。
3.  **强制下线**：
    *   C端用户再次发起任何需要鉴权的请求时，`Middleware` 发现 Token 已被吊销，返回 `401 Unauthorized`。
    *   C端 APP 拦截到 `401` 状态码，自动清空本地缓存并跳转至登录页。
