# 综合新闻 APP - 数据库表结构设计字典

## 1. 数据库设计规范 (Best Practices)
本项目后端采用 `Node.js` + 关系型数据库 `MySQL 8.0`（或兼容版本）。结合行业最佳实践，数据库设计遵循以下规范：
1. **字符集与排序规则**：统一采用 `utf8mb4` 和 `utf8mb4_general_ci`，全面支持 Emoji 表情（评论、动态中常用）。
2. **基础字段标配**：每张业务表必须包含 `id` (主键)、`create_time` (创建时间)、`update_time` (更新时间)、`is_deleted` (逻辑删除标识，0正常 1删除)。
3. **主键策略**：采用分布式 ID（雪花算法 Snowflake 生成的 `BIGINT`），避免使用自增主键带来的数据量猜测和分库分表困难。
4. **命名规范**：表名和字段名全部小写，使用下划线 `_` 分隔（如 `user_info`）。

## 2. 核心业务表设计

### 2.1 用户表 (`users`)
存储系统用户的核心注册信息与鉴权数据。

| 字段名 | 数据类型 | 长度 | 主键/外键 | 允许空 | 默认值 | 备注说明 |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| `id` | BIGINT | 20 | PK | 否 | - | 用户全局唯一ID (雪花算法) |
| `phone` | VARCHAR | 20 | - | 否 | - | 手机号码，做 UNIQUE 唯一索引 |
| `password` | VARCHAR | 128 | - | 否 | - | 登录密码 (Bcrypt/Argon2 单向哈希) |
| `nickname` | VARCHAR | 50 | - | 否 | - | 用户昵称 |
| `avatar_url` | VARCHAR | 255 | - | 是 | NULL | 头像的 OSS/CDN 地址 |
| `bio` | VARCHAR | 200 | - | 是 | NULL | 个人简介/个性签名 |
| `status` | TINYINT | 4 | - | 否 | 1 | 账号状态 (1:正常, 2:封禁, 3:注销) |
| `create_time` | DATETIME | - | - | 否 | CURRENT_TIMESTAMP | 注册时间 |
| `update_time` | DATETIME | - | - | 否 | CURRENT_TIMESTAMP | 最后更新时间 (ON UPDATE CURRENT_TIMESTAMP) |
| `is_deleted` | TINYINT | 1 | - | 否 | 0 | 逻辑删除标记 (0:未删, 1:已删) |

### 2.2 文章/新闻表 (`articles`)
存储平台发布的图文资讯。

| 字段名 | 数据类型 | 长度 | 主键/外键 | 允许空 | 默认值 | 备注说明 |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| `id` | BIGINT | 20 | PK | 否 | - | 文章唯一ID |
| `author_id` | BIGINT | 20 | FK | 否 | - | 发布者ID (关联 `users.id`)，做普通索引 |
| `category_id` | INT | 11 | - | 否 | - | 分类ID (如 1:科技, 2:娱乐等) |
| `title` | VARCHAR | 128 | - | 否 | - | 文章标题 |
| `cover_urls` | JSON | - | - | 是 | NULL | 封面图数组（支持无图、单图、三图等多种信息流排版布局） |
| `content` | LONGTEXT | - | - | 否 | - | 文章富文本内容 (支持HTML带`<img src="...">`或块级JSON，实现图文并茂) |
| `view_count` | INT | 11 | - | 否 | 0 | 浏览量/阅读数 |
| `like_count` | INT | 11 | - | 否 | 0 | 点赞数统计冗余字段 |
| `comment_count`| INT | 11 | - | 否 | 0 | 评论数统计冗余字段 |
| `status` | TINYINT | 4 | - | 否 | 1 | 状态 (0:草稿, 1:已发布, 2:下架/审核不通过) |
| `create_time` | DATETIME | - | - | 否 | CURRENT_TIMESTAMP | 创建时间 |
| `update_time` | DATETIME | - | - | 否 | CURRENT_TIMESTAMP | 更新时间 |
| `is_deleted` | TINYINT | 1 | - | 否 | 0 | 逻辑删除标记 |

### 2.4 评论表 (`comments`)
统一管理文章、视频、动态下的用户评论，采用**拉链式设计**支持嵌套回复。

| 字段名 | 数据类型 | 长度 | 主键/外键 | 允许空 | 默认值 | 备注说明 |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| `id` | BIGINT | 20 | PK | 否 | - | 评论唯一ID |
| `user_id` | BIGINT | 20 | FK | 否 | - | 评论者ID |
| `target_id` | BIGINT | 20 | - | 否 | - | 目标主体ID (文章ID或视频ID)，做普通索引 |
| `target_type` | TINYINT | 4 | - | 否 | - | 目标类型 (1:文章, 2:视频, 3:动态) |
| `parent_id` | BIGINT | 20 | - | 否 | 0 | 父评论ID (0表示顶级评论，非0表示回复某条评论) |
| `content` | VARCHAR | 1000| - | 否 | - | 评论纯文本内容 |
| `like_count` | INT | 11 | - | 否 | 0 | 评论被点赞数 |
| `create_time` | DATETIME | - | - | 否 | CURRENT_TIMESTAMP | 评论时间 |
| `is_deleted` | TINYINT | 1 | - | 否 | 0 | 逻辑删除标记 (用户删除或后台审核删除) |

### 2.5 用户互动关联表 (`user_interactions`)
记录用户的点赞、收藏行为。这是一种高频写入表，实际生产中可先写入 Redis，再异步同步至此表。

| 字段名 | 数据类型 | 长度 | 主键/外键 | 允许空 | 默认值 | 备注说明 |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| `id` | BIGINT | 20 | PK | 否 | - | 记录ID |
| `user_id` | BIGINT | 20 | FK | 否 | - | 行为发起者ID |
| `target_id` | BIGINT | 20 | - | 否 | - | 被互动的目标ID (文章/视频/评论ID) |
| `target_type` | TINYINT | 4 | - | 否 | - | 目标类型 (1:文章, 2:视频, 3:评论) |
| `action_type` | TINYINT | 4 | - | 否 | - | 行为类型 (1:点赞, 2:收藏) |
| `create_time` | DATETIME | - | - | 否 | CURRENT_TIMESTAMP | 行为发生时间 |

> **索引建议**：对 `(user_id, target_id, target_type, action_type)` 建立**联合唯一索引 (Unique Index)**，防止同一个用户对同一篇文章重复点赞。

## 3. 索引优化策略 (Index Strategy)
为了应对综合新闻 APP 的检索需求，后端在建表时应注意：
1. **分页性能**：新闻列表页查询极为频繁。应在 `articles` 表的 `(category_id, status, create_time)` 字段上建立**联合索引**，加速分类新闻的按时间倒序拉取。
2. **用户维度查询**：在 `articles` 和 `videos` 表的 `author_id` 字段建立索引，加速“我的发布”页面的查询。
3. **全文检索**：由于 MySQL 的 `LIKE '%keyword%'` 性能较差，对于新闻标题和内容的搜索，后期可引入 **Elasticsearch (ES)** 或 **MeiliSearch**。若仅使用 MySQL，可在 `title` 和 `content` 字段建立 `FULLTEXT` 全文索引（需 MySQL 5.7+ 且配置 ngram 解析器支持中文）。
